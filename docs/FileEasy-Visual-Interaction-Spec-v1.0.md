# FileEasy Visual Interaction Spec v1.0

## 1. 文档信息

- 产品名：`FileEasy`
- 文档类型：`视觉交互设计规范`
- 版本：`v1.0`
- 状态：`首版整理`
- 上游文档：[FileEasy-PRD-v1.0.md](/Users/jainaluo/易思态/code/Xplay/docs/FileEasy-PRD-v1.0.md)
- 适用范围：
  - APK 端首页
  - 扫码上传页
  - FileEasy v1 不包含远程 `/admin` 管理页

## 2. 文档目标

本规范用于在不超出 PRD 范围的前提下，统一 FileEasy v1 的视觉语言、信息层级、页面结构、交互规则和状态反馈方式，作为后续 UI 设计、前端实现和验收对齐依据。

本规范只解决以下问题：

- 页面看起来应当是什么气质
- 信息应该如何分层呈现
- 核心操作应该如何被引导
- 登录、上传、续传、空间不足等关键动作如何反馈
- 手机浏览器、桌面浏览器和 APK 首页如何保持一致体验

本规范不包含以下内容：

- 技术架构设计
- 接口字段定义
- 服务端模块拆分
- 超出 PRD 的新增功能

## 3. 设计定位

### 3.1 产品气质

FileEasy 的视觉与交互气质应统一为：

- `极简`：只保留完成上传所需的信息
- `可信`：让用户明确知道文件会上传到哪台设备、当前是否可用
- `稳定`：对长时任务、网络波动、空间不足给出清晰反馈
- `轻工具化`：像一个可靠的小型局域网文件服务，而不是播放器、系统工具箱或运维面板

### 3.2 设计关键词

- 清晰
- 克制
- 稳定
- 可恢复
- 低学习成本

### 3.3 交互原则

1. 先让用户看到页面，再要求密码验证，符合 PRD 的“先看页后验证”规则。
2. 所有长流程都必须可感知，尤其是上传、续传、网络恢复。
3. 所有阻断错误都必须清楚说明影响范围，尤其是空间不足、文件超限、类型不支持。
4. 所有异常都必须说明“发生了什么、当前结果是什么、用户下一步能做什么”。
5. 视觉优先级始终让位于主任务：扫码、登录、上传。

## 4. 统一视觉语言

### 4.1 色彩系统

建议采用“深海蓝 + 青绿色”作为主视觉方向，避免偏娱乐化、播放器化或过度技术面板化。

色彩角色建议如下：

| 角色 | 建议色值 | 用途 |
|---|---|---|
| 主色 Primary | `#1565C0` | 主按钮、链接、焦点、高优先级入口 |
| 强调色 Accent | `#00A896` | 上传进度、成功态、积极状态 |
| 成功色 Success | `#1B8F5A` | 成功提示、可用状态 |
| 警告色 Warning | `#C67A00` | 空间不足预警、网络波动提示 |
| 危险色 Danger | `#C53B32` | 删除、失败、不可恢复错误 |
| 主文字 | `#1F2430` | 标题、正文 |
| 次文字 | `#5B6575` | 说明文案、元信息 |
| 弱文字 | `#8A93A3` | 占位、辅助文案 |
| 背景色 | `#F6F8FB` | 页面背景 |
| 卡片底色 | `#FFFFFF` | 卡片、弹层、列表单元 |
| 边框色 | `#E3E8EF` | 分隔线、输入框、卡片描边 |

使用原则：

- 成功、警告、危险色只用于状态表达，不作为页面主风格大面积使用。
- APK 端与 Web 端共用同一套状态色语义，避免同一状态在两端颜色含义不一致。
- 上传进度优先使用 `Primary -> Accent` 的连续表达，避免使用危险色做进度高亮。

### 4.2 字体系统

中文建议字体栈：

- `MiSans`
- `Noto Sans SC`
- `PingFang SC`
- `HarmonyOS Sans SC`
- `sans-serif`

数字、IP 地址、文件大小、百分比建议使用等宽数字或局部等宽字体呈现，保证扫描效率。

字号层级建议：

- 页面主标题：`28/32`
- 模块标题：`20/24`
- 卡片标题：`16/20`
- 正文：`14/20` 或 `16/24`
- 辅助说明：`12/18`
- 数据标签：`12/16`

### 4.3 版式与间距

统一采用 `8px` 间距体系，关键间距建议如下：

- 页面外边距：`16px`、`24px`、`32px`
- 卡片内边距：`16px` 或 `20px`
- 组件间垂直间距：`12px`、`16px`
- 大区块间距：`24px`、`32px`
- 触控目标最小高度：`44px`

### 4.4 圆角与阴影

- 输入框、按钮：`12px`
- 卡片：`16px`
- 弹窗、抽屉：`20px`

阴影应轻量，仅用于抬升卡片层级，不做夸张悬浮感。整体更接近稳定工具，而不是营销页面。

### 4.5 图标与插图

- 图标采用线性或轻填充风格，笔画简洁。
- 页面不依赖装饰性插画建立风格，重点靠排版、留白和状态色建立秩序。
- 文件类型图标只作为辅助识别，不喧宾夺主。

### 4.6 动效原则

- 只保留必要动效：状态切换、进度变化、弹层出现、列表加载。
- 动效时长建议 `160ms` 到 `240ms`。
- 上传进度必须平滑更新，但不能通过过强动画制造“虚假忙碌感”。
- 危险操作确认层出现时，应比普通弹层更稳重，避免轻佻动画。

## 5. 统一交互规则

### 5.1 按钮优先级

- 主按钮：当前页面唯一最重要动作，如“登录”“选择文件”
- 次按钮：辅助动作，如“取消”
- 文字按钮：低干扰动作，如“查看支持格式”

同一区域中不应同时出现两个同等视觉权重的主按钮。

### 5.2 输入规则

- 输入框默认展示简短明确的占位说明。
- 密码输入默认隐藏，支持显隐切换。

### 5.3 反馈规则

- 成功反馈：简短明确，不打断主流程
- 失败反馈：明确说明失败原因和影响范围
- 进行中反馈：说明当前进度与是否可继续等待
- 空状态反馈：说明当前没有内容，不暗示系统故障

### 5.4 确认机制

- 修改密码：提交前无需额外确认，但提交成功后需明确提示登录态失效逻辑

### 5.5 响应式原则

- 手机端优先单列布局
- 桌面端可使用双列或三段式布局
- 关键信息顺序在所有终端保持一致：标题、状态、主操作、详细信息

## 6. 信息架构

### 6.1 APK 端

- 首页
  - 服务状态
  - 上传二维码
  - 上传地址
  - 修改密码入口

### 6.2 扫码端

- 上传页
  - 页面头部
  - 登录验证区
  - 上传操作区
  - 上传任务列表
  - 异常与状态提示
- `/admin`
  - 不属于 FileEasy v1 远程页面
  - 不设计远程文件列表、预览、重命名、删除、批量删除、下载

## 7. APK 首页视觉交互规范

### 7.1 页面目标

APK 首页只负责回答 4 个问题：

1. 服务现在是否可用
2. 其他设备应该扫哪里
3. 其他设备应该访问什么地址
4. 密码在哪里修改

### 7.2 页面结构

首页建议采用单页卡片化布局，自上而下分为 4 个区块：

1. 顶部信息区
2. 状态区
3. 二维码与地址主卡区
4. 操作区

推荐结构：

- 顶部信息区：产品名 `FileEasy` + 一句简短说明
- 状态区：当前服务状态标签
- 主卡区：二维码为视觉中心，地址为次中心
- 操作区：`修改密码`

### 7.3 视觉重点

- 二维码是首页绝对主视觉中心，应处于页面视觉焦点位置。
- 上传地址应与二维码形成强关联，便于“扫码失败时手输地址”。
- 服务状态应显著但不压过二维码，建议以状态条或状态徽标方式呈现。
- 修改密码入口应明确可见，但视觉权重低于扫码上传主任务。

### 7.4 状态设计

首页至少覆盖以下状态：

| 状态 | 文案方向 | 视觉建议 |
|---|---|---|
| 服务正常 | `服务已开启` | 使用成功色标签 |
| 服务启动中 | `服务启动中，请稍候` | 使用中性色 + 轻微加载态 |
| 当前无可用局域网地址 | `未检测到可用局域网地址` | 使用警告色提示，隐藏二维码 |
| 服务异常 | `服务异常，暂时不可访问` | 使用危险色提示 |

说明要求：

- 状态文案要面向普通用户，不使用技术术语。
- 出现异常时，不展示复杂日志，只提示结果与下一步动作。

### 7.5 二维码区规范

- 二维码区使用白底高对比容器，保证扫码成功率。
- 二维码下方必须直接展示上传地址文本。
- 地址文本支持换行但不应被截断到无法识别。
- 若无可用地址，则二维码区显示“当前无法生成上传二维码”。

### 7.6 地址展示规范

- 地址文字建议使用更易识别数字的字重和字体。
- 地址区应强调“这是上传页地址”。
- 不展示远程管理页地址，避免普通上传用户误认为 v1 提供文件管理入口。

### 7.7 操作区规范

- `修改密码` 为次级操作
- 入口使用卡片按钮或大号列表项呈现，保证中老年或远距离操作时也易点按

### 7.8 适配规则

- 手机竖屏：单列布局，二维码居中
- 平板或横屏盒子：二维码与地址可上下居中，操作区固定在下方
- 远距离展示场景下，应优先保证二维码尺寸与地址可读性，而不是追求信息密度

## 8. 扫码上传页视觉交互规范

### 8.1 页面目标

上传页只负责让用户顺畅完成三件事：

1. 输入密码进入上传
2. 选择单文件或多文件上传
3. 明确感知上传状态、失败原因与续传结果

### 8.2 页面结构

上传页建议分为以下区块：

1. 页面头部
2. 登录验证区
3. 上传操作区
4. 上传列表区
5. 状态提示区

### 8.3 登录前页面状态

根据 PRD，用户进入上传页后应先看到页面，再进行密码验证。因此登录前状态不应是纯空白拦截页，而应是“可见页面 + 登录层”模式。

建议表现：

- 页面头部正常可见
- 上传主区域以禁用态或半遮罩态呈现
- 登录卡片置于首屏中心位置
- 文案清晰说明“输入密码后开始上传文件”

### 8.4 登录区规范

- 登录框仅保留一个密码输入主字段
- 主按钮为 `登录`
- 密码错误时在输入框下就近提示，不使用全局模糊报错
- 登录成功后平滑切换到上传区，避免整页闪烁重载

### 8.5 上传操作区规范

- 首屏主按钮为 `选择文件`
- 次文案明确支持“单文件 / 多文件上传”
- 上传区可采用大面积拖拽/点选卡片，但移动端仍以点选文件为主
- 支持格式说明应简洁展示，不铺满大段文字

建议在上传区持续展示以下辅助信息：

- 单文件最大 `4GB`
- 支持断点续传
- 支持多文件上传

### 8.6 上传任务列表规范

每个上传任务至少展示：

- 文件名
- 文件大小
- 当前状态
- 进度百分比
- 进度条

状态建议统一为：

- `等待上传`
- `上传中`
- `上传中断`
- `正在恢复`
- `上传完成`
- `上传失败`

说明要求：

- 多文件场景下，列表应优先保证文件名可识别，而不是展示过多列信息。
- 上传中的文件项应有明显进度表达，完成项应收敛为低干扰成功态。

### 8.7 续传状态规范

断点续传是 v1 的信任核心，交互上必须被明确表达。

建议规则：

- 页面刷新或中断恢复后，若任务可继续，展示 `已恢复上传任务`
- 正在恢复时，任务状态显示 `正在恢复`
- 超过 `24 小时`导致续传状态失效时，提示 `上传任务已过期，请重新选择文件`

### 8.8 错误状态规范

上传页至少覆盖以下错误：

| 场景 | 提示方向 |
|---|---|
| 密码错误 | 密码错误，请重新输入 |
| 文件类型不支持 | 当前文件类型不支持上传 |
| 单文件超过 4GB | 文件超过单文件大小限制 |
| 存储空间不足，无法新建上传 | 设备空间不足，请联系管理员清理 |
| 上传过程中空间耗尽 | 上传失败，设备空间不足 |
| 网络中断 | 网络中断，正在等待恢复 |
| 续传状态过期 | 上传任务已过期，请重新上传 |

错误提示原则：

- 优先说人话，不说接口码
- 优先说明结果，再说明建议动作
- 就近提示优先于全局打断

### 8.9 手机端布局规范

- 采用单列布局
- 登录卡片、上传卡片、任务列表纵向排列
- 任务项使用上下信息堆叠，避免过窄导致文件名不可读
- 底部留足安全区域，避免按钮贴边

### 8.10 桌面端布局规范

- 可采用双段式布局
- 左侧为上传区，右侧为任务列表
- 或上方操作区、下方任务表格区
- 桌面端允许展示更完整的文件大小、状态、更新时间等辅助信息

## 9. `/admin` 与远程文件管理移出范围

### 9.1 v1 路由边界

- `/` 是 FileEasy v1 唯一远程用户页面。
- `/admin` 不再是 FileEasy v1 页面，不应设计为远程管理页。
- 如果浏览器访问 `/admin`，可展示 `404`、`410` 或回到上传页，但不得出现文件管理能力。

### 9.2 v1 不设计的能力

以下能力不属于本视觉规范的 v1 交付范围：

- 远程文件列表
- 远程文件搜索
- 远程在线预览
- 远程重命名
- 远程下载
- 远程删除
- 远程批量删除

### 9.3 后续设计口径

后续如需文件管理能力，应作为 v1.1 或独立需求重新补齐信息架构、权限边界、视觉交互与验收标准。

## 10. 文案与提示规范

### 10.1 文案风格

- 面向普通用户
- 句子短
- 少术语
- 少命令口吻
- 少感叹号

推荐语气：

- `输入密码后开始上传文件`
- `上传完成`
- `网络中断，正在等待恢复`
- `设备空间不足，请联系管理员清理`

避免语气：

- `鉴权失败`
- `服务器异常`
- `Token 失效`
- `分片合并失败`

### 10.2 按钮文案

优先使用动词直达型文案：

- `登录`
- `选择文件`
- `修改密码`

## 11. 状态反馈规范

### 11.1 全局状态层级

- 轻提示：成功、复制类、非阻断结果
- 就地提示：输入错误、局部失败
- 模块提示：上传区错误、空间不足、网络恢复中
- 弹窗提示：修改密码等 APK 本机确认或结果反馈

### 11.2 加载状态

- 初始加载显示骨架或轻量占位
- 上传中以进度为主，不额外叠加强旋转动画

### 11.3 成功状态

- 成功提示停留时间短，不遮挡主内容
- 上传完成后，完成项保留可识别成功状态，便于用户确认

## 12. 适配与一致性要求

### 12.1 APK 与 Web 的一致性

- 共用相同状态色语义
- 共用相同主按钮风格
- 共用相同文案语气
- 共用相同危险操作确认逻辑

### 12.2 手机与桌面的一致性

- 页面结构顺序一致
- 功能入口命名一致
- 状态文案一致
- 仅布局方式变化，不改变核心任务路径

## 13. 验收口径

若页面设计满足以下条件，则可视为符合本视觉交互规范：

- APK 首页一眼能看懂服务状态、二维码、上传地址和修改密码入口
- APK 首页不出现远程管理页入口
- 上传页登录前已可见页面结构，且登录动作不突兀
- 上传页对单文件、多文件、上传中、上传完成、上传失败、续传恢复均有清晰表达
- `/admin` 不作为 v1 远程文件管理页验收
- 手机浏览器与桌面浏览器均能完成完整流程
- 所有错误提示都清楚、克制、可执行
- 页面整体气质符合“极简、可信、稳定”的 FileEasy 定位

## 14. 后续产出建议

基于本规范，下一步可继续拆出以下设计资产：

1. 页面线框图
2. 高保真视觉稿
3. 组件清单
4. 前端实现规范
5. 文案状态清单

以上后续资产必须继续以本规范和 PRD 为边界，不额外扩需求。
