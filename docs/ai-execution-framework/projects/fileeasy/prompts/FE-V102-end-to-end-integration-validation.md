## 你需要完成的工作

### 1. 按整体验收清单逐项验证

重点核对以下几类场景：

#### APK 层
- FileEasy 首页是否仍然是纯服务端壳
- 服务状态是否正确
- 地址与二维码逻辑是否正确

#### 上传页
- 是否先展示页面，再要求登录
- 登录逻辑是否正确
- 多文件上传是否成立
- 进度和状态展示是否成立
- 续传提示是否成立
- 错误提示是否明确

#### 上传后端
- 是否仍有白名单和 `4GB` 边界
- 分片状态恢复是否成立
- 同名自动重命名是否成立
- 是否仍然不存在旧上传入口绕过

#### 管理页
- 列表是否成立
- 搜索是否成立
- 预览是否按支持范围成立
- 重命名是否是真实能力
- 下载 / 单删 / 批量删除是否成立
- 是否仍然没有批量下载、日志页、监控页等越界功能

### 2. 记录通过项与失败项

最终必须明确输出：

#### 通过项
- 哪些场景已经成立
- 为什么判断它成立

#### 失败项
- 哪些场景没有成立
- 失败现象是什么
- 是否会阻塞 FileEasy v1 进入交付收口

### 3. 区分“发布阻塞”与“非阻塞问题”

#### 发布阻塞
例如：

- 主流程断裂
- 上传链路不能完成
- 管理页关键能力不能用
- 产品边界明显超出或不符合 PRD

#### 非阻塞问题
例如：

- 文案细节
- 轻微 UI 排版问题
- 次要交互体验问题

不要混写。

### 4. 给 PM 一个明确判断

你必须在最后给出下面二选一建议：

#### 选项 A
- 可以进入发布阻塞项收口阶段

#### 选项 B
- 当前还不能进入发布收口
- 先需要补这些关键验证或修这些关键问题：`...`

## 推荐验证方式

你可以结合下面这些方式完成判断：

1. 代码路径检查
2. 页面结构检查
3. 接口路径检查
4. 本地构建验证
5. 如果条件允许，可补浏览器页面实际走查

如果某些场景因为环境受限无法实测，必须明确写出来，不能假设通过。

## 最低验证要求

至少要覆盖下面这些判断：

1. 上传页是否仍符合“先展示后认证”
2. 上传页是否仍有多文件、逐文件进度、续传状态
3. 上传后端是否仍有：
- 白名单
- `4GB` 限制
- 分片状态恢复
- 同名自动重命名

4. 管理页是否仍有：
- 列表
- 搜索
- 预览
- 重命名
- 下载
- 单删
- 批量删除

5. 是否仍然没有：
- 批量下载
- 日志页
- 监控页
- 超出 PRD 的后台功能
