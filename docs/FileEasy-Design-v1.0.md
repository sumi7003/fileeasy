# FileEasy Design v1.2

## 1. 文档信息

- 产品名：`FileEasy`
- 版本：`v1.2`
- 状态：`定稿候选版`
- 上游文档：[FileEasy-PRD-v1.0.md](/Users/jainaluo/易思态/code/Xplay/docs/FileEasy-PRD-v1.0.md)
- 本版说明：
  - 按 `FE-CR001` 移除 FileEasy v1 远程 `/admin` 管理页
  - 明确远程 Web 表面仅保留上传页 `/`
  - 补齐纯服务端隔离闭环
  - 补齐热点 / Wi-Fi 地址与二维码策略
  - 补齐开机自启与前台服务生命周期设计
  - 补齐文件白名单上传校验
  - 固化同名上传自动重命名落盘规则
  - 删除超出 PRD 的日志查询接口
  - 收口远程改密权限边界
  - 补齐断点续传分片状态模型
  - 补齐 4GB 文件上限服务端校验

## 2. 设计目标

本设计文档用于明确 FileEasy v1 的技术形态、模块边界、页面结构、核心接口、运行生命周期和异常处理策略，作为后续 AI 实施与验收的唯一设计依据。

## 3. 总体架构

FileEasy 采用单 APK 本地服务架构：

1. Android APK 作为宿主应用。
2. APK 内嵌本地 HTTP 服务。
3. Web 上传页 `/` 由 APK 本地服务提供。
4. 所有文件保存在 Android 本地存储中。
5. 局域网内其他终端通过浏览器访问该服务。

## 4. 关键产品决策

### 4.1 密码修改后的登录态

- 管理员修改密码后，所有已有登录态立即失效。
- 上传页必须重新登录。

### 4.2 正式文件持久化策略

- v1 正式文件使用 APK 私有目录。
- 接受“应用卸载后文件丢失”的行为。
- v1 不引入外部公开目录持久化方案。

## 5. 纯服务端隔离方案

### 5.1 产物形态

- 基于现有 `apps/android-player` 工程扩展独立 `fileeasy` flavor。
- 现阶段不单独拆新仓库。
- 通过 flavor 隔离应用名、包名、产品文案和功能开关。

### 5.2 v1 目标

- `fileeasy` 与 `xplay` 可分别构建。
- `fileeasy` 交付物是纯服务端 APK。
- `fileeasy` 不得暴露播放器产品能力。
- `fileeasy` 的运行、页面、文案、启动链路不得依赖播放器逻辑成立。

### 5.3 v1 纯服务端禁用矩阵

| 维度 | `xplay` | `fileeasy` v1 要求 |
|---|---|---|
| App 名称 | Xplay | FileEasy |
| 包名 | `com.xplay.player` | 独立 `applicationId` |
| 首页入口 | 控制中心 + 播放器角色 | 仅文件服务首页 |
| 播放器开关 | 可见 | 不可见 |
| 播放全屏入口 | 可见 | 不存在 |
| 播放器状态文案 | 可见 | 不存在 |
| 播放器相关页面路由 | 保留 | 不暴露 |
| 播放器启动链路 | 保留 | 不自动启动 |
| 播放器服务依赖 | 保留 | 不作为主流程前置条件 |
| 产品文案 | Xplay 体系 | FileEasy 体系 |

### 5.4 实施边界

- v1 允许复用现有本地服务、Web 资源加载、基础存储和认证能力。
- v1 不允许把“关闭一个按钮”视为隔离完成。
- v1 实现必须满足：
  - 首页不出现播放器相关文案与入口
  - 不要求用户配置播放相关内容
  - 不依赖播放列表、设备注册、播放器连接状态

## 6. 运行时模块划分

### 6.1 APK 端模块

- `App Shell`
  - 首页
  - 密码修改入口
  - 服务状态展示
  - 上传地址与二维码展示
- `Lifecycle Manager`
  - 开机自启
  - 前台服务拉起
  - 异常重启兜底
- `Local Server`
  - 路由注册
  - 登录态处理
  - 上传接口
  - FileEasy 模式下不暴露远程文件管理接口
- `Storage Manager`
  - 文件落盘
  - 分片临时文件管理
  - 存储空间检查
  - 元数据维护
- `Auth Manager`
  - 默认密码
  - 密码修改
  - 登录态 cookie / token
  - 登录态失效广播
- `Log Manager`
  - 上传记录
- `Network Address Manager`
  - 当前可访问地址选择
  - 地址变化监听
  - 二维码内容更新

### 6.2 Web 端模块

- `Upload Page`
  - 登录验证
  - 多文件选择
  - 分片上传
  - 上传进度与续传状态
- `Admin Page`
  - 不属于 FileEasy v1 远程 Web 范围
  - `/admin` 不得渲染远程文件管理页
  - 远程文件列表、预览、重命名、删除、批量删除、下载均不得作为 v1 能力暴露

## 7. 页面信息架构

### 7.1 APK 首页

首页仅包含以下信息和动作：

- 当前服务状态
- 上传页二维码
- 上传地址
- 修改密码

不加入以下内容：

- 系统监控面板
- 日志中心
- 容量分析图
- 复杂说明区
- 播放器角色区
- 播放控制区

### 7.2 上传页

上传页包含：

- 页面标题
- 密码验证区
- 上传文件区
- 上传列表区
- 上传进度区
- 上传失败提示
- 续传状态提示

### 7.3 远程管理页边界

FileEasy v1 不包含远程管理页。

如后续需要文件管理，应作为 v1.1 或独立需求重新设计，不得复用旧 v1 验收项直接扩回当前范围。

## 8. 地址选择与二维码策略

### 8.1 地址选择目标

APK 首页展示的上传地址和二维码，必须尽可能指向当前局域网内其他设备可访问的地址。

### 8.2 地址选择优先级

按以下优先级选择 host：

1. 当前活动 Wi-Fi 网络的 IPv4 地址
2. 当前活动热点共享网络的 IPv4 地址
3. 其他可用的非回环 IPv4 地址
4. 若无可用地址，则视为“当前不可访问”

### 8.3 地址生成规则

- 地址格式固定为 `http://{host}:3000/`
- 二维码内容使用上传页地址

### 8.4 刷新时机

以下时机必须刷新地址和二维码：

- App 启动
- 本地服务启动成功
- Wi-Fi / 热点连接变化
- IP 地址变化
- 前台页面恢复可见时

### 8.5 无可用地址时的行为

- 首页显示“当前未检测到可用局域网地址”
- 不展示可扫码二维码
- APK 本机密码修改入口仍可保留

## 9. 路由与接口设计

### 9.1 Web 页面路由

- `/`：上传页
- `/admin`：不属于 FileEasy v1 页面路由；可返回 `404`、`410` 或重定向到 `/`，但不得暴露远程文件管理能力

### 9.2 认证相关接口

- `POST /api/v1/auth/login`
  - 登录
- `POST /api/v1/auth/logout`
  - 退出登录

说明：

- v1 不开放通用远程改密接口给局域网页面用户。
- 修改密码能力仅允许由 APK 本地入口触发。
- 如系统内部保留改密接口，也只能服务 APK 本地入口，不作为公共 HTTP 能力开放给远程已登录用户。

### 9.3 上传相关接口

- `GET /api/v1/ping`
  - 服务可达性检查
- `GET /api/v1/home/summary`
  - 上传页或 APK 首页所需摘要
- `POST /api/v1/upload/init`
  - 初始化上传任务
- `POST /api/v1/upload/chunk`
  - 上传分片
- `POST /api/v1/upload/complete`
  - 合并并完成上传
- `GET /api/v1/upload/status/{uploadId}`
  - 查询续传状态
- `DELETE /api/v1/upload/status/{uploadId}`
  - 取消未完成上传

#### 9.3.1 `upload/init` 约束

- 服务端必须校验 `fileSize <= 4GB`
- 服务端内部文件大小统一使用 `Long`
- 文件类型必须先通过白名单校验
- 超过 4GB 时返回固定错误：`单文件不能超过 4GB`

### 9.4 远程文件管理接口边界

FileEasy 模式下不应暴露远程文件管理接口，包括：

- `GET /api/v1/files`
- `GET /api/v1/files/{id}`
- `GET /api/v1/files/{id}/preview`
- `GET /api/v1/files/{id}/download`
- `PATCH /api/v1/files/{id}`
- `DELETE /api/v1/files/{id}`
- `POST /api/v1/files/batch-delete`
- `/uploads/{path...}` 远程下载面

这些接口如在共享服务中保留给其他 flavor，也必须在 FileEasy 模式下禁用或不可达。

## 10. 数据模型设计

### 10.1 文件元数据

字段建议：

- `id`
- `originalName`
- `displayName`
- `storedFilename`
- `extension`
- `mimeType`
- `size`
- `category`
- `createdAt`
- `updatedAt`

### 10.2 上传会话

字段建议：

- `uploadId`
- `fileName`
- `fileSize`
- `chunkSize`
- `totalChunks`
- `uploadedChunks`
- `uploadedChunkIndexes`
- `status`
- `expiresAt`
- `createdAt`

说明：

- `uploadedChunks` 仅作为统计字段存在。
- 断点续传恢复必须依赖 `uploadedChunkIndexes` 或等价位图结构。
- 不能只依赖已上传计数来判断缺失分片。

### 10.3 操作日志

字段建议：

- `id`
- `fileId`
- `action`
- `ip`
- `result`
- `createdAt`

## 11. 分片上传与断点续传方案

### 11.1 方案选择

采用服务端控制的分片上传方案：

- 浏览器端按 `8MB` 切片
- 每个文件先创建 `uploadId`
- 每片上传时携带：
  - `uploadId`
  - `chunkIndex`
  - `totalChunks`
  - `fileName`
  - `fileSize`
- 服务端记录已上传分片
- 断开后可按状态继续上传缺失分片
- 服务端需持久化每个已上传分片索引

### 11.2 生命周期

- 上传任务初始化后生成会话
- 未完成会话保留 `24 小时`
- 超时后自动删除分片状态和临时文件

### 11.3 状态恢复

- `GET /api/v1/upload/status/{uploadId}` 至少返回：
  - `totalChunks`
  - `uploadedChunkIndexes`
  - `missingChunkIndexes`
  - `status`
- `missingChunkIndexes` 可由服务端基于 `totalChunks` 与 `uploadedChunkIndexes` 动态计算
- 该接口必须支持乱序上传、重复上传和缺片恢复场景

### 11.4 合并策略

- 所有分片上传完成后，调用完成接口
- 服务端按顺序合并分片
- 合并前必须校验：
  - 是否所有分片已齐全
  - 累计大小是否与声明的 `fileSize` 一致
- 合并成功后生成正式文件元数据
- 合并完成、准备正式落盘时再确定最终显示名称
- 临时分片文件清理

### 11.5 同名上传自动重命名

- 同名上传自动重命名规则在 `upload/complete` 阶段执行
- 上传初始化阶段不锁定最终 `displayName`
- 服务端在正式落盘前检查目标目录是否已存在同名文件
- 若存在同名文件，则自动生成安全名，如 `文件(1).ext`
- 最终返回给前端的文件名以正式落盘后的 `displayName` 为准

## 12. 文件存储设计

### 12.1 正式文件目录

- 使用 APK 私有目录下的正式文件存储区
- v1 接受应用卸载后正式文件一并删除

### 12.2 临时分片目录

- 单独的临时上传目录
- 按 `uploadId` 隔离

### 12.3 空间控制

- 上传初始化前检查剩余空间
- 分片写入前再次检查
- 合并前再次检查
- 空间不足时终止任务并返回明确错误

## 13. 文件类型上传准入策略

### 13.1 v1 白名单

允许上传的扩展名仅限：

- 文档：`pdf doc docx xls xlsx ppt pptx txt`
- 图片：`jpg jpeg png gif webp`
- 视频：`mp4 mov`
- 音频：`mp3 wav m4a`
- 压缩文件：`zip`

### 13.2 校验规则

上传接口必须执行双重校验：

1. 基于扩展名的 allowlist 校验
2. 基于 MIME 的基础一致性校验

若扩展名与 MIME 明显冲突，以服务端规则判定为不支持类型。

### 13.3 错误返回

命中不支持类型时：

- 拒绝创建正式上传任务
- 返回明确错误：`当前文件类型不在 FileEasy v1 支持范围内`

## 14. 远程文件预览与下载边界

FileEasy v1 不提供远程预览与远程下载。

允许上传的文件类型仍按白名单执行，但白名单只代表“允许接收”，不代表远程可预览或可下载。

后续如需预览、下载或文件列表能力，必须重新补齐：

- 权限边界
- API 设计
- 页面设计
- 验收清单

## 15. 认证与登录态设计

### 15.1 密码模型

- 当前版本仅保留一套密码
- 默认密码首次可直接使用
- 管理员可在 APK 中修改密码
- 管理员修改密码操作仅允许通过 APK 本地入口完成

### 15.2 登录态

- 登录成功后签发登录态
- 有效期 7 天
- 登录态仅用于上传页远程能力

### 15.3 密码修改后的处理

- 修改密码后，立即废弃所有现有登录态
- 所有已登录浏览器必须重新登录

### 15.4 页面访问规则

- 访问上传页：允许先进入页面，再要求登录
- 访问 `/admin`：不得进入远程文件管理页
- 调用上传相关接口：必须校验登录态
- 即使已登录，远程网页用户也不能获得改密能力

## 16. 同名上传自动重命名规则

### 16.1 远程重命名边界

FileEasy v1 不提供远程重命名接口或重命名页面。

### 16.2 上传冲突处理

- 若上传目标文件名冲突，沿用自动重命名规则
- 例如：`文件(1).ext`

### 16.3 上传同名文件处理

- 上传链路中的同名文件不报冲突失败
- 上传链路中的同名文件不覆盖旧文件
- 最终行为统一为自动重命名

## 17. 日志设计

记录以下行为：

- 上传成功

日志能力在 v1 只要求“记录”，不要求提供日志查询页面或日志查询接口。

日志基础字段：

- 时间
- IP
- 文件标识
- 文件名
- 操作类型
- 操作结果

## 18. 开机自启与前台服务生命周期

### 18.1 开机自启

- 通过 `BootReceiver` 监听开机完成广播
- 开机后尝试拉起本地服务

### 18.2 前台服务常驻

- 本地服务以 Android 前台服务方式运行
- 必须有持续通知
- 通知至少包含：
  - FileEasy 服务名称
  - 当前运行状态

### 18.3 拉起时机

以下时机应确保服务可用：

- App 首次启动
- 用户主动开启服务
- 开机完成后

### 18.4 异常恢复

- 若服务意外退出，应按系统允许的方式尝试恢复
- 若恢复失败，首页应能反映“服务未运行”状态

### 18.5 验收口径

v1 验收至少覆盖：

- 安装后手动启动服务成功
- 设备重启后服务可恢复
- 服务运行时前台通知存在

## 19. 异常处理设计

### 19.1 空间不足

- 禁止新上传
- 上传中失败并提示清理空间

### 19.2 分片缺失

- 完成上传时若发现缺片，返回缺失列表

### 19.3 会话过期

- 返回“上传任务已过期，请重新上传”

### 19.4 非法文件名

- 返回明确错误，不写入正式文件

### 19.5 不支持文件类型

- 返回白名单限制错误
- 不进入正式存储

### 19.6 文件大小超限

- 当 `fileSize > 4GB` 时，在 `upload/init` 阶段直接拒绝
- 返回固定错误：`单文件不能超过 4GB`

## 20. 首版不做

- 播放器链路改造
- 自动发现机制
- 公网穿透
- 文档在线预览
- 远程 `/admin` 管理页
- 远程文件列表
- 远程文件预览
- 远程文件重命名
- 远程文件删除
- 远程文件批量删除
- 远程文件下载
- 批量下载
- 复杂系统首页
- 日志查询页
- 日志查询接口

## 21. AI 实施约束

- 开发必须遵循 PRD 范围。
- 不允许在没有更新 PRD 和设计文档的情况下新增功能。
- 后续所有任务派发、交付检查和测试场景确认，都以本设计文档为准。
